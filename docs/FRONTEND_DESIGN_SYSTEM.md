# FunPay Cloud Frontend Design System

Версия: 2.0  
Статус: Production baseline (single source of truth)  
Владелец: Frontend команда FunPay Cloud  
Источник визуального эталона: актуальный лендинг `/`

---

## 1. Назначение

Этот документ фиксирует **единый фирменный стиль**, требования к оформлению компонентов и правила реализации UI во всех сегментах продукта:
- `landing` (`/`),
- `platform` (`/platform/*`),
- `blog` (`/blog/*`),
- `docs` (`/docs/*`),
- `admin` (`/admin/*`).

Цель: чтобы любой дизайнер, разработчик или AI-агент мог открыть **один файл** и принять корректные визуальные/UX решения без догадок.

---

## 2. Дизайн-манифест бренда

### 2.1 Характер

FunPay Cloud визуально должен считываться как:
- надежный B2B SaaS,
- быстрый, структурный и понятный,
- технологичный без “гиковского шума”,
- премиальный без агрессивного визуального перегруза.

### 2.2 Впечатление при первом экране

Пользователь должен почувствовать:
- «сервис продуманный и аккуратный»,
- «сюда можно доверить деньги и автоматизацию»,
- «интерфейс современный, но не сложный».

### 2.3 Антипаттерны (запрещено)

- случайные цвета и “разовые” стили;
- разные радиусы/тени внутри одного паттерна;
- мелкий низкоконтрастный текст в ключевых сценариях;
- “карточки-клоны” без визуальной иерархии;
- визуальный шум ради “красиво”.

---

## 3. Дизайн-принципы (обязательные)

1. **Consistency first**
- Сначала системные токены и готовые паттерны, потом новые решения.

2. **Hierarchy over decoration**
- Иерархия важнее декоративных эффектов.

3. **Semantic tokens only**
- Никаких raw-цветов в компонентах, кроме отдельно согласованных исключений.

4. **Mobile as production surface**
- Мобилка — не “обрезанная версия”, а полноценный сценарий.

5. **Accessibility by default**
- Контраст, фокус, клавиатурная навигация и ARIA обязательны.

6. **Data clarity for operations UI**
- На платформах управления (чаты/заказы/лоты) приоритет на читаемость данных.

---

## 4. Архитектура тем и scope-изоляция

### 4.1 Scope-и

- `.landing` — светлая брендовая тема главной страницы.
- `.platform` (или текущий платформенный scope) — темная операционная тема.
- `.blog` — отдельная тематизация блога (light/dark переключение в рамках блога).
- `.admin` — темный системный контур админки.

### 4.2 Правило изоляции

- Любая страница использует **только свои токены scope**.
- Нельзя “подмешивать” цвета/переменные другого scope.
- Общие компоненты используют семантические токены `--ds-*`, которые мапятся внутри scope.

## 5. Token System (единый стандарт)

## 5.1 Брендовые foundation-токены (эталон landing)

```css
.landing {
  --bg: #FBFAF7;
  --bg-2: #F3F1EB;
  --ink: #0E1116;
  --ink-2: #2A2F38;
  --muted: #6B7280;
  --line: #E7E3DA;
  --line-2: #D9D4C6;
  --card: #FFFFFF;
  --accent: #3A2FE0;
  --accent-ink: #FFFFFF;
  --accent-soft: #EEECFE;
  --good: #0E8A57;
  --warn: #B86E00;
  --bad: #B23A3A;
}
```

## 5.2 Семантические DS-токены (цель для всех разделов)

```css
:root {
  --ds-bg-primary: ;
  --ds-bg-secondary: ;
  --ds-surface: ;
  --ds-surface-elevated: ;
  --ds-border-default: ;
  --ds-border-strong: ;

  --ds-text-primary: ;
  --ds-text-secondary: ;
  --ds-text-muted: ;
  --ds-text-inverse: ;

  --ds-accent: ;
  --ds-accent-hover: ;
  --ds-accent-soft: ;
  --ds-success: ;
  --ds-warning: ;
  --ds-danger: ;

  --ds-focus-ring: ;
}
```

### 5.3 Token Governance

- Новый токен добавляется только после ревью.
- Один токен = одна роль.
- Запрещено использовать success/warning/danger в brand-ролях.
- Изменение токена обязано сопровождаться скриншотами регрессии ключевых экранов.

## 6. Типографика

### 6.1 Шрифты

- UI: `Inter`
- Mono: `JetBrains Mono`
- Fallback: `-apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif`

### 6.2 Шкала текста

- Display/Hero: `34–64px`
- H1: `32–48px`
- H2: `26–40px`
- H3: `20–28px`
- Body: `14–16px`
- Secondary: `12–13px`
- Meta/Caption: `10–12px`

### 6.3 Правила набора

- Заголовки: плотные, с контролируемым `letter-spacing`.
- Текст интерфейса: максимальная читаемость, избегать слишком светлых оттенков.
- Технические значения: `JetBrains Mono` (цены, id, статусы системы при необходимости).

---

## 7. Spacing, Grid, Radius, Shadow

### 7.1 Базовая spacing-сетка

- База: `4px`
- Ключевые шаги: `4 / 8 / 12 / 16 / 20 / 24 / 32 / 40 / 48 / 64 / 80 / 96`

### 7.2 Контейнеры

- Max content width landing: `1240px`
- Стандарт page content платформы: `1200–1320px` (в зависимости от раздела)
- Горизонтальные поля mobile: минимум `16px`

### 7.3 Скругления

- Контролы: `8–10px`
- Карточки: `12–16px`
- Крупные секции/баннеры: `18–24px`
- Pills: `999px`

### 7.4 Тени

- Surface low: мягкая, почти незаметная
- Surface mid: карточки hover
- Surface high: hero/промо/модалки

Тени должны усиливать иерархию, не превращать UI в “глянцевую витрину”.

---

## 8. Breakpoints и адаптив

Базовые диапазоны:
- Mobile: `<640px`
- Tablet: `640–1024px`
- Desktop: `>1024px`

Дополнительно для больших экранов:
- Wide desktop: `>=1440px`

### 8.1 Обязательные правила адаптива

- Touch target минимум `44x44`.
- Критические actions всегда остаются доступны на mobile.
- Сложные таблицы на mobile: horizontal scroll wrapper + фиксированная min-width.
- Для двухколоночных desktop-сценариев обязателен fallback в одну колонку на mobile.

---

## 9. Иконография и графика

- Иконки: `lucide-react`, единая stroke-стилистика.
- Размеры по умолчанию: `14 / 16 / 18 / 20`.
- Иконка не заменяет текст в критичных действиях без `aria-label`.
- Брендовые логотипы использовать только из approved ассетов:
  - `logo_full_new.svg`
  - `logo_short_new.svg`

---

## 10. Состояния интерфейса

Каждый интерактивный компонент обязан иметь:
- default,
- hover,
- active/pressed,
- focus-visible,
- disabled,
- loading (если действие сетевое),
- error/success (если применимо).

Отсутствие хотя бы одного критичного состояния = незавершенный компонент.

---

## 11. Компонентные стандарты (production)

## 11.1 Buttons

Варианты:
- `ghost`
- `outline`
- `primary`
- `accent`

Размеры:
- `default`
- `lg`

Требования:
- единая высота в рамках размера,
- предсказуемые hover/focus,
- disabled визуально и функционально блокирует действие,
- иконка внутри кнопки выравнивается по baseline текста.

## 11.2 Inputs / Select / Textarea

- Фон и border через токены.
- Focus-state всегда заметный.
- Placeholder менее контрастный, чем текст.
- Ошибки отображаются рядом с контролом и не требуют hover для прочтения.

## 11.3 Toggles/Switches

Стандартный размер:
- трек `44x24`,
- круг `20x20`,
- отступы и трансляция без вылета за пределы.

Единая реализация по всему проекту, без локальных модификаций геометрии.

## 11.4 Cards

- Внутренняя вертикальная ритмика строго по spacing-токенам.
- У карточки должен быть один смысловой центр (title + primary metric/action).
- Нельзя смешивать 3+ визуально равносильных акцента в одной карточке.

## 11.5 Data Table

- Header контрастнее body.
- Числовые значения выравниваются последовательно.
- Статусы имеют единые color semantics.
- На mobile — горизонтальный скролл или карточный fallback.

## 11.6 Sidebar

- Активный пункт: читаемый высокий контраст + четкий active-индикатор.
- Неактивные пункты: не “мертвые”, но вторичные.
- Все иконки приведены к одному размеру и optical alignment.

## 11.7 Topbar

- Высота: стабильная (`56px` для платформы, если не оговорено отдельно).
- В topbar только функционально важные блоки.
- Dropdown-меню обязаны иметь непрозрачный фон и читаемый border.

## 11.8 Modal / Drawer

- Overlay обязателен.
- Закрытие: по клику вне + `Esc` + крестик.
- Контентная структура: title, description, body, actions.
- destructive action выделяется опасным стилем и требует подтверждения.

## 11.9 Notifications / Toasts

- Короткий текст: причина + действие/следующий шаг.
- Error не исчезает слишком быстро.
- Success/info могут автозакрываться.

## 11.10 FAQ Accordion

- Заголовочная колонка всегда выравнивается по левой границе контента.
- На mobile FAQ-блок центрируется только по контейнеру, но текст остаётся с левым выравниванием.

---

## 12. Паттерны страниц

## 12.1 Landing

- Светлая брендовая тема.
- Явная секционная иерархия.
- CTA на `/auth/register`, secondary на `/auth/login`.
- Footer содержит disclaimer о неаффилированности с FunPay.

## 12.2 Platform

- Темная операционная тема.
- Приоритет data density и скорости чтения.
- Минимум визуального шума в ключевых сценариях (чаты/заказы/лоты).

## 12.3 Blog

- Контент-first.
- Высокая читаемость типографики.
- SEO/OG/структурированные данные обязательны.

## 12.4 Docs

- Технический тон.
- Левый nav + контент + optional TOC.
- Кодовые блоки с моноширинным шрифтом и четким контрастом.

## 12.5 Admin

- Функциональность и мониторинг.
- Ровные таблицы, фильтры, пагинация, цветовая семантика логов.

---

## 13. Motion & Micro-interaction

- Базовые transition: `120–240ms`, `ease`/`ease-out`.
- Hover-анимации не должны влиять на layout и вызывать reflow.
- Автокарусели: пауза при hover/focus и доступное ручное управление.
- Избегать агрессивных bounce/zoom эффектов.

---

## 14. Accessibility Checklist

Перед merge обязательно проверить:
- контраст AA;
- `focus-visible` для кнопок, ссылок, инпутов;
- aria-label у иконок-кнопок;
- корректная иерархия заголовков;
- корректная таб-навигация;
- без ловушек фокуса в модалках/drawer.

---

## 15. Performance Guidelines

- Изображения: только `next/image` с `width/height/sizes`.
- Шрифты: `next/font`.
- Избегать лишнего клиентского JS в контентных блоках.
- Визуальные эффекты не должны ломать mobile FPS.

---

## 16. Контент и microcopy

- Тон: уверенный, дружелюбный, без маркетинговой агрессии.
- Action-кнопки: глагольные и короткие (`Открыть`, `Сохранить`, `Продлить`).
- Сообщения об ошибках: что случилось + что делать дальше.
- Технические термины объясняются, если пользователь массовый.

---

## 17. Реализация в коде (Tailwind + tokens)

### 17.1 Общее правило

- Сначала utility-классы через семантические переменные.
- Если паттерн повторяется 3+ раза — вынос в компонент.

### 17.2 Когда можно inline style

Разрешено только для:
- динамических градиентов,
- runtime width/progress,
- случаев, где utility-класс выразить нельзя без потери ясности.

Все остальное — через классы и токены.

## 18. Миграция на единый стиль (план релиза)

## Phase 1 — Foundation
- Утвердить и заморозить токены.
- Привязать все scope к семантическому слою.

## Phase 2 — Shared primitives
- Выровнять `Button`, `Card`, `Input`, `Badge`, `Toggle`, `Modal`, `Drawer`, `Table`.

## Phase 3 — Страницы
- Порядок миграции:
1. `/platform/dashboard`
2. `/platform/accounts`
3. `/platform/chats`
4. `/platform/lots` + `/platform/warehouse`
5. `/blog`
6. `/docs`
7. `/admin`

## Phase 4 — Hardening
- Удаление legacy CSS.
- Visual regression snapshots.
- Финальная полировка spacing/типографики.

---

## 19. Definition of Done для UI-задачи

UI-задача считается завершенной, если:
- соблюдены правила этого документа;
- есть responsive-поведение для mobile/tablet/desktop;
- все интерактивные состояния реализованы;
- проведена проверка на контраст и keyboard flow;
- `npm run build` проходит;
- нет визуальных регрессий в соседних разделах.

---

## 20. QA-чеклист перед релизом

1. Визуальная проверка на `320/375/768/1024/1280/1440`.
2. Проверка hover/focus/disabled/loading состояний.
3. Проверка длинных текстов и локализации.
4. Проверка переполнения таблиц и карточек.
5. Проверка dropdown/modal/drawer overlay и layering.
6. Проверка тёмной/светлой темы для сегментов, где она предусмотрена.
7. Smoke-проверка ключевых сценариев (auth, chats, orders, settings).

---

## 21. Governance и процесс изменений

- Любое изменение брендовых токенов или ключевых компонентных правил оформляется PR к этому файлу.
- PR должен включать:
  - мотивацию,
  - список затронутых экранов,
  - before/after скриншоты,
  - план миграции (если требуется).
- Обратная совместимость для существующих разделов обязательна, если не согласован массовый редизайн.

---

## 22. Быстрый старт для нового агента/дизайнера

Перед любой UI-правкой:
1. Прочитать этот документ полностью.
2. Определить scope (`landing/platform/blog/docs/admin`).
3. Проверить, есть ли готовый паттерн нужного блока.
4. Использовать токены и существующие компоненты.
5. Проверить адаптив и состояния.
6. Прогнать build и сделать визуальную самопроверку.

---

## 23. Юридический/брендовый дисклеймер

В публичных страницах (landing/footer, при необходимости docs) использовать формулировку:

> FunPay Cloud не аффилирован с FunPay и не является официальным инструментом FunPay.

Формулировка может меняться редакционно, но смысл должен сохраняться.
